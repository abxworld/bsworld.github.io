运算符

  • 与(&)运算符

    转换为二进制的两个数，进行与操作，相同位置全部为1才为1 ，否则为0

  • 或( | )运算符

    转换为二进制的两个数，进行或操作，相同位置只要有一个是1，则为1，否则为0

  • 非( ~ )

    转换为二进制的一个数，进行非操作，如果为1则转为0，如果是0则转为1

  • 异或( ^ )

    转换为二进制的连个数，进行异或操作，相同位置相同则为0，不同则为1，任何数与0做异或操作都等于该数自己，任何数与1做异或操作，都相当于该数做非操作

HashMap寻找key的过程

  • 对key进行hash操作，获取到相应的hash值后

  • 对第一步生成的hash值,再散列到整个数组上

    伪代码如下:

1
2

int hash = hash(key);
int i = indexFor(key, table.length);

先看一下hash方法:

1
2
3
4

   static final int hash(Object key) {
        int h;
        return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
    }

首先是取key的hashCode，然后再将hashCode值(int类型,共32位)与向右位移16位的hashCode做异或处理

indexFor方法伪代码:

1
2
3

static int indexFor(int h, int length){
     return h & (length - 1);
 }

indexFor代码,正好解释了为什么HashMap的数组长度要取2的整次幂,因为这样（数组长度-1）正好相当于一个“低位掩码”,“与”操作的结果就是散列值的高位全部归零，只保留低位值，用来做数组下标访问

数组长度要取2的整次幂的好处

  • 与操作符相比于 % 的效率更高(但是结果值与和%的值是不一样)

  • 与操作符的含义是，全部为1结果才为1，其余为0，现在(数组长度-1)低位全部为1，那么在key进行hash扰动后进行与操作，key的hash如果是均匀的那么进行与操作也是均匀的

  • 进行与操作后的值在 0 至 数组长度-1 之间（左右皆为闭区间）

以初始长度16为例，16-1=15，2进制表示是00000000 00000000 00001111。

和某散列值做“与”操作如下，结果就是截取了最低的四位值。

但这时候问题就来了，这样就算我的散列值分布再松散，要是只取最后几位的话，碰撞也会很严重。更要命的是如果散列本身做得不好，分布上成等差数列的漏洞，恰好使最后几个低位呈现规律性重复，就无比蛋疼。

​ 这时候“扰动函数”的价值就体现出来了，说到这里大家应该猜出来了

看下面这个图:

​ 右位移16位，正好是32bit的一半，自己的高半区和低半区做异或，就是为了混合原始哈希码的高位和低位，以此来加大低位的随机性。而且混合后的低位掺杂了高位的部分特征，这样高位的信息也被变相保留下来。

为什么在进行扰动的时候采用异或的方式?

  • 首先异或的定义是两位相同则为0，不同则为1，看下图:

    从图中可以看出，在hash出现均匀的情况下，最终结果出现0或者1的概率是相同的，而高低位混合，肯定是需要0或者1出现的概率是一样的，所以采用异或这种方式

JDK 7做了4次右移，估计是边际效应的原因，JDK8就只做了一次右移。

另外 JDK8在链表长度超过8的时候，就使用红黑树做存储。这一改变大大优化了很多性能。

负载因子(默认是0.75)

它的作用的hashmap扩容的阈值，假如说负载因子是0.75，初始数组长度是16，那儿数组在使用超过12的时候就会执行扩容

如果负载因子过小:

  • ​ hashmap频繁扩容，并且有更多的内存得不到有效利用

负载因子过大:

  • hash冲突的可能性大大增加，极端情况下如果负载因子为1，那么只有等数组的所用空间都被使用的时候才会发生扩容，但是这个时候hash散列也会激烈的冲突

默认负载因子为什么是0.75

  • 时间和空间综合考量的结果，负载因子大于0.75的时候，空间得到更充分的利用，但是hash冲突的可能性达达增加，并且大概率会使用链表红黑树等，查询速度降低，小于0.75，减少hash冲突，查询速度加快
